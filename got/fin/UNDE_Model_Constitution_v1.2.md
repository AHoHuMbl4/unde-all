# UNDE Financial Model — Конституция
### v1.2 | Шаг 0 — фундамент перед любыми расчётами

---

## 0. Зачем этот документ

Каждая ячейка модели должна опираться на единые определения. Без этого:
- разные листы трактуют «MAU» по-разному
- формула выручки содержит скрытые допущения
- фазовые переходы привязаны к датам, а не к реальности

Конституция фиксирует **три вещи**: что мы считаем, как работает валюта, когда переключаются фазы.

## 1. ОПРЕДЕЛЕНИЯ

### 1.1. Пользовательские метрики

| Термин | Определение | Scope | Примечание |
|--------|------------|-------|------------|
| **Onsite MAU** | Уникальный пользователь, совершивший ≥1 Onsite Active Visit за календарный месяц | Per Local Seed | **Базовая метрика для монетизации и порогов Local Seed.** НЕ глобально. Пользователь, активный в 2 моллах = 2 Onsite MAU в разных Seeds |
| **Total MAU** | Уникальный пользователь, совершивший ≥1 Active Visit (Onsite ИЛИ Remote) за календарный месяц | Per Local Seed | Для retention-аналитики и полноты картины. Не используется в порогах Local Seed |
| **Onsite Active Visit** | Сессия в приложении с подтверждённым физическим присутствием в кластере (indoor positioning) | Per Local Seed | База для Verified Actions и ценности для Provider. Минимальная длительность: [H] >60 сек |
| **Remote Active Visit** | Сессия взаимодействия с каталогом кластера без физического присутствия (browsing из дома/офиса) | Per Local Seed | Полезно для retention и «подготовки» визита, но монетизируется иначе. Не входит в Onsite MAU |
| **Recommendation** | Одна подборка вариантов, сформированная AI в ответ на запрос пользователя | Per session | Один запрос = одна Recommendation, даже если содержит несколько товаров |
| **Provider Touchpoint** | Факт появления уникального Provider в Recommendation пользователя | Per Recommendation | Метрика покрытия и здоровья сети. Один Provider = один Touchpoint per Recommendation, независимо от количества товаров |
| **Item Impression** | Одна товарная позиция (слот), показанная в Recommendation | Per Recommendation | Метрика монетизации. Credits тратятся на приоритет слотов (item-level). Один Provider может иметь несколько Item Impressions в одной Recommendation |
| **D7 / D30 Retention** | % пользователей, совершивших Onsite Active Visit на день 7 / 30 после первого Onsite Active Visit | Per Local Seed | Считается по кластеру, не глобально. Привязан к Onsite, не к Remote |

### 1.2. Provider-метрики

| Термин | Определение | Scope | Примечание |
|--------|------------|-------|------------|
| **Provider** | Любой бизнес, подключённый к UNDE с активным Real-Time State Feed | Универсально | На старте: fashion-ритейлер в ТЦ. Позже: салон, ресторан, сервис |
| **Active Provider** | Provider с актуальным каталогом (обновление ≤7 дней) И ≥1 Touchpoint за месяц | Per Local Seed | Просто зарегистрированный ≠ Active |
| **Real-Time State Feed** | Данные о текущем состоянии Provider: наличие, цены, размеры, промо, свободные слоты | Per Provider per Seed | Актуальность = ключевой компонент UNDE Index |
| **UNDE Index** | Единый показатель вовлечённости Provider в экосистему. Считается per Provider per Local Seed | Per Provider per Seed | Нельзя купить. Только заслужить. Минимальный Index = допуск к Перформансу намерения |

**Компоненты UNDE Index (§4.2 Monetization Strategy). Все веса — [H]:**

| Компонент | Примерный вес [H] | Что измеряет |
|-----------|-------------------|--------------|
| Привлечение | ~40% | Сколько activated пользователей привёл Provider |
| Retention | ~25% | D30 retention привлечённых пользователей |
| Качество данных | ~15% | Актуальность и полнота Real-Time State Feed |
| Конверсия | ~20% | % рекомендаций → подтверждённое действие |

### 1.3. Действия и верификация

| Термин | Определение | Класс верификации |
|--------|------------|-------------------|
| **Verified Action** | Универсальная единица подтверждённого результата | C1–C6 (см. ниже) |
| **Agent-Verified Action (АПД)** | Подтверждённый результат, достигнутый при участии агента UNDE | Весит больше обычного VA в Index |

**Лестница верификации (§4.3):**

| Класс | Метод | Надёжность | Доступность |
|-------|-------|-----------|-------------|
| C1 | Цифровое намерение (показ → клик «Где купить») | Низкая | Day 1 |
| C2 | Близость (нахождение в зоне Provider >3 мин) | Средняя | Day 1 |
| C3 | Самоотчёт («Я купил!» + бонус аватару) | Средняя | Day 1 |
| C4 | Скан чека (фото → распознавание) | Высокая | MVP+ |
| C5 | QR/промокод (скан стикера в магазине) | Высокая | По договору |
| C6 | Интеграция с кассой (POS) | Максимальная | Глубокое партнёрство |

**Правило для модели:** На Фазе 1 базовая модель строится на C1–C3. C4–C6 — сценарный слой (upside), не baseline.

### 1.4. Единица роста

| Термин | Определение |
|--------|------------|
| **Local Seed** | Минимальная жизнеспособная сеть в локальном кластере. Все пороги выполнены одновременно (см. §3.1) |
| **Кластер** | Физическое пространство принятия решений: ТЦ, район, lifestyle-зона. На старте: ТЦ |

## 2. CREDITS ECONOMY — ПРАВИЛА ВАЛЮТЫ

### 2.1. Фундаментальный принцип

> **Earned Credits ≠ выручка.**
> Earned Credits = гейт (право). Выручка = только Purchased Credits.

Формула:

```
Max_Purchase = Earned_Credits × K
Revenue_Credits = Purchased_Credits × Price_Per_Credit
```

Где `Purchased_Credits ≤ Max_Purchase`

### 2.2. Зарабатывание Credits

Credits начисляются Provider'у только за **качественных** пользователей.
Все значения — **[H] гипотезы дизайна механики**, будут калиброваться по пилотным данным:

| Действие пользователя | Credits | Тип | Почему это ценно |
|----------------------|---------|-----|-----------------|
| Установил app по ссылке Provider | +10 | [H] | Минимальный сигнал интереса |
| Прошёл онбординг + первая примерка | +100 | [H] | Активированный пользователь |
| Вернулся на D7 | +50 | [H] | Формируется привычка |
| Вернулся на D30 | +150 | [H] | Лояльный пользователь |
| Поделился образом | +30 | [H] | Виральность |
| Подтверждённое действие (покупка) | +200 | [H] | Конверсия = главная ценность |
| АПД: резерв/бронь через агента | +250 | [H] | Агент довёл до результата |
| АПД: покупка/оплата через агента | +400 | [H] | Максимальный outcome |

**Антифрод:** установка = ничтожно мало (+10). Реальные Credits — за activation и retention.

### 2.3. K-коэффициент (Earn-to-Spend)

Все значения K и пороги — **[H] гипотезы**, будут уточняться по willingness-to-pay тестам:

| Заработано Credits | K [H] | Можно докупить [D] | Итого доступно [D] |
|-------------------|-------|-------------------|-------------------|
| 0 | — | 0 | 0 |
| 1,000 | 0.3 | 300 | 1,300 |
| 5,000 | 0.5 | 2,500 | 7,500 |
| 20,000 | 0.7 | 14,000 | 34,000 |
| 50,000+ | 1.0 | 50,000 | 100,000 |

**Ключевое:** нулевой баланс = нулевой доступ. Нельзя прийти с деньгами и «купить рекламу».

### 2.4. На что тратятся Credits

**Core: Перформанс намерения (Intent Performance)**
- AI подбирает релевантные варианты для пользователя
- Provider вкладывает X Credits → его вариант получает приоритет среди релевантных
- Конкурент вкладывает больше → получает более высокий приоритет
- Нерелевантный Provider не получит доступ ни за какие Credits

**Условие:** минимальный UNDE Index для траты Credits на приоритет.

### 2.5. Price Per Credit

**Гипотеза для Фазы 2:** $0.10 per Credit [H] (из §7.2 Monetization Strategy).

**Для модели:** это переменная, не константа. Будет уточняться по willingness-to-pay тестам в Фазе 1.

### 2.6. Правила модели для Credits

| Правило | Описание |
|---------|----------|
| **Earned ≠ Revenue** | Earned Credits — это право (гейт). Выручка только от Purchased |
| **K — не скидка** | K — это потолок покупки, не мультипликатор выручки |
| **Scope = per Seed** | Credits зарабатываются и тратятся в рамках конкретного Local Seed |
| **Сгорание** | [H] Credits действительны 90 дней с момента последней квалифицирующей активности. Затем: decay 10% баланса/неделю. Квалифицирующая активность: привлечение retained пользователей (D7+), обновление Feed, verified actions. «Просто зайти в дашборд» — не квалифицирует |
| **Перенос** | [T] Гипотеза: Credits не переносятся между Seeds для сетей (будет уточнено) |

## 3. ПОРОГИ LOCAL SEED

### 3.1. Определение жизнеспособности

Local Seed = **все** пороги выполнены **одновременно** в одном кластере.
Все пороги — **[H] гипотезы** до появления пилотных данных:

| Метрика | Порог | Почему |
|---------|-------|--------|
| Onsite MAU в кластере | 500+ [H] | Критическая масса для данных и виральности |
| Active Providers | 10+ [H] | Достаточно вариантов для качественных рекомендаций |
| k_combined | > 1.0 [H] | Сеть растёт органически (сумма k_user + k_provider) |
| D30 retention | > 40% [H] | Ритуал формируется |
| Покрытие категории | > 50% fashion-магазинов [H] | Рекомендации полезны, а не фрагментарны |

**Декомпозиция k-factor (критично для Блока 3):**

| Метрика | Определение | Драйвер | Примечание |
|---------|------------|---------|------------|
| **k_user** | Коэффициент user→user виральности: сколько новых activated пользователей приводит один существующий через шеринг, рекомендации, UGC | Маховик доверия | Органический рост продукта |
| **k_provider** | Коэффициент provider-driven acquisition: сколько новых activated пользователей приходит через каналы Provider'ов (deep links, QR, соцсети Provider) | Маховик Earn-to-Spend | Ядро модели. Гипотеза: 30–40% трафика [H] |
| **k_combined** | k_user + k_provider | Оба маховика | Порог Local Seed: > 1.0 |

**Формулы (помесячно, считать только activated = дошедших до Onsite Active Visit):**

```
k_user     = New_Activated_Users_from_User_Sharing  / Existing_Onsite_MAU   (per month)
k_provider = New_Activated_Users_from_Provider_Channels / Existing_Onsite_MAU   (per month)
k_combined = k_user + k_provider
```

**Activated** = совершил Onsite Active Visit (не просто установил app). Это антифрод: фейковые установки без визита не считаются.

**Почему два k:** Смешивание в один скрывает, какой маховик работает. Если k_combined > 1.0, но k_user = 0.1 и k_provider = 0.95 — это Provider-зависимая модель (рискованнее). Если k_user = 0.6 и k_provider = 0.5 — сбалансированный рост (устойчивее). VC должен видеть оба.

### 3.2. Почему локальные метрики

- «Много установок» может означать «по чуть-чуть везде» = нет критической массы нигде
- Сетевые эффекты работают локально: пользователь ценит UNDE в СВОЁМ молле
- Provider платит за трафик в СВОЁМ молле, не глобально
- Все метрики считаются для каждого кластера отдельно
- **Onsite MAU** (не Total MAU) используется в порогах — Remote browsing не создаёт ценность для Provider

### 4.1. Принцип

> **Фазы привязаны к порогам сети, не к календарю.**
> Время — ориентир, не триггер.

### 4.2. Фаза 1: Earn-only

| Параметр | Значение |
|----------|---------|
| **Триггер входа** | Старт |
| **Триггер выхода** | Local Seed достигнут (все пороги §3.1) |

| **Агентный слой** | Только L1 (вместе выбираем) |

### 4.3. Фаза 2: Earn-to-Buy

| Параметр | Значение |
|----------|---------|
| **Триггер входа** | Local Seed достигнут в первом кластере |

| Параметр | Значение |
|----------|---------|

| **Фокус** | Масштабирование прибыли + отраслевой стандарт |

## 5. REVENUE STREAMS — Core vs Accelerators

### 5.1. Core: Intent Performance (Credits)

Единственный обязательный revenue stream. Модель должна быть жизнеспособна **только на этом**.

```
Revenue_Core = Purchased_Credits × Price_Per_Credit
```

Где:
- `Purchased_Credits ≤ Earned_Credits × K`
- `Price_Per_Credit` = $0.10 [H] (гипотеза, Фаза 2)
- Credits тратятся на приоритет слотов (Item Impressions) среди релевантных

### 5.2. Accelerator 1: Аналитика и атрибуция

Улучшает ARPU per Provider, но Core должен сходиться без этого.

- Базовая аналитика входит в пакет Core (бесплатно)
- Расширенная: когортные отчёты, конкурентная разведка, оптимизация ассортимента
- Доступ привязан к объёму Credits и UNDE Index, не к фиксированной подписке
- **Ориентир:** [V] В публичных контрактах Placer.ai встречаются annual fees ~$14K–$60K/год (в зависимости от объёма, сущностей, срока — данные из публичных order forms, не официальный прайс). [B] Нижний бенчмарк: StoreTraffic ~$25/мес/устройство — другой класс продукта, но показывает готовность платить за footfall-аналитику вообще

### 5.3. Accelerator 2: Enterprise (операторы пространств)

Канал дистрибуции + revenue, не отдельный бизнес.

- Агрегированная аналитика по всему ТЦ
- Сравнительные отчёты по UNDE Index между арендаторами
- Инструменты оптимизации состава арендаторов
- **Модель:** enterprise-контракты (Фаза 3)

### 5.4. Для модели: Revenue Evolution

| Фаза | Core (Credits) | Accelerator 1 (Analytics) | Accelerator 2 (Enterprise) |
|-------|---------------|--------------------------|---------------------------|
| Earn-only | $0 | $0 | $0 |
| Earn-to-Buy | Purchased × $0.10 | Пилот | $0 |
| Full Stack | Purchased × $0.10 + Перформанс результата | Допродажа по Index | Enterprise контракты |

## 6. EVIDENCE & SOURCES — КЛАССИФИКАЦИЯ

Каждая цифра в модели должна иметь тип источника:

| Тип | Обозначение | Что это | Пример |
|-----|------------|---------|--------|
| **V** (Verified) | [V] | Внешний проверенный источник с ссылкой | Placer.ai публичные контракты ~$14K–$60K/год |
| **B** (Benchmarked) | [B] | Бенчмарк из индустрии, но не прямой аналог | StoreTraffic ~$25/мес/устройство |
| **H** (Hypothesis) | [H] | Внутренняя гипотеза, требует валидации | Price Per Credit = $0.10 |
| **D** (Derived) | [D] | Расчётная цифра из других параметров модели | ARPU = f(Credits, K, adoption) |
| **T** (TBD) | [T] | Ещё не определено, placeholder | Сгорание Credits, срок |

**Правило:** ни одна ячейка модели не может содержать число без квадратных скобок с типом. Если тип не указан — это баг.

## 7. ОРГАНИЧЕСКИЙ ПОРОГ — ЗАЩИТА ПОЛЬЗОВАТЕЛЯ

> **Гипотеза [H]:** в каждой Recommendation не более 70% слотов (Item Impressions) могут быть приоритизированы за Credits. Минимум 30% слотов — чисто органические.
> Источник: §3.3 Monetization Strategy (адаптировано с уровня «рекомендаций» на уровень «слотов»).

**Почему слоты, а не рекомендации:**
Рекомендация может содержать 5 или 20 товаров. «3 из 10 рекомендаций органические» — двусмысленно: одна «органическая» с 3 товарами может компенсировать десять «платных» по 20. Привязка к слотам делает порог однозначным при любом размере подборки.

**Что это значит для модели:**
- В каждой Recommendation: `Monetizable_Slots ≤ Total_Slots × 0.70` [H]
- Минимум 30% слотов — AI решает без влияния Credits
- Это **архитектурный потолок монетизации**, не бизнес-решение

**Для расчётов:**
```
Monetizable_Item_Impressions = Total_Item_Impressions × (1 - Organic_Threshold)
Organic_Threshold = 0.30 [H]
```

## 8. ТРИ МАХОВИКА — МЕХАНИКА ДЛЯ БЛОКА 3

Для справки при переходе к Network Dynamics:

**Маховик доверия (пользовательский):**
Лучше попали в рекомендацию → больше доверия → чаще спрашивает → больше сигналов → ещё точнее

**Маховик покрытия (данные мира):**
Больше Provider'ов → полнее Feed → точнее рекомендации → выше конверсия → конкуренты вынуждены подключаться

**Маховик Earn-to-Spend (рост + выручка):**
Provider приводит пользователей → зарабатывает Credits → тратит на приоритет → видит окупаемость → приводит ещё больше

## 9. КОНТРОЛЬНЫЙ ЧЕКЛИСТ

Перед любым расчётом в модели — проверить:

- [ ] MAU = Onsite MAU (с позиционированием) для монетизации и порогов? Remote MAU отдельно?
- [ ] Provider = Active (актуальный каталог + Provider Touchpoints), не просто зарегистрированный?
- [ ] Монетизация привязана к Item Impressions (слоты), а покрытие — к Provider Touchpoints?
- [ ] Revenue = Purchased Credits × Price, НЕ Earned × K × Price?
- [ ] Verified Actions $3–10 живут в сценарном слое, не в baseline?
- [ ] Фаза определяется метрикой, не датой?

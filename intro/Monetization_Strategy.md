**UNDE**

Стратегия монетизации и роста

Внутренний документ | Февраль 2026 | v4.1 (draft)

> **О цифрах в документе.** Все параметры Credits, коэффициенты K, пороги Local Seed и примерные цены — это **продуктовые гипотезы**, подлежащие валидации в пилоте. Внешние ориентиры (retail media, аналитика посещаемости, стоимость визита) отмечены ссылками и проверены по публичным источникам. # 1. Философия монетизации

## 1.1. Главный принцип

**UNDE растёт за счёт бизнеса. Бизнес приводит нам пользователей, чтобы получить право продвигаться.**

Это инверсия классической модели. Не мы платим за привлечение — бизнес платит нам трафиком за доступ к экосистеме.

| Классическая модель | Модель UNDE |
| :---- | :---- |
| Платформа платит за трафик | Бизнес приводит трафик платформе |
| Бизнес платит за размещение | Бизнес зарабатывает право быть видимым |
| Транзакционные отношения | Экосистемная зависимость |
| Конкуренты копируют функции | Конкуренты не могут скопировать зависимость |

## 1.2. Моральный контракт

UNDE — на стороне пользователя. Всегда. Любая монетизация подчиняется моральному контракту:

- Мы не продаём внимание пользователя. Мы усиливаем релевантность.
- Мы не продаём персональные данные. Никому. Никогда.
- Мы не берём комиссию с транзакций пользователя. UNDE — не маркетплейс.
- Если решение о монетизации противоречит интересам пользователя — мы выбираем пользователя.

Каждое решение о монетизации проверяется вопросом: **«Это укрепляет или разрушает ритуал?»** Если разрушает — не делаем, какой бы ни была выручка.

## 1.3. Принцип релевантности (и почему это не манипуляция)

Пользователь всегда получает ценность без рекламы. Продвижение происходит ТОЛЬКО среди релевантных альтернатив (по стилю, бюджету, размеру, контексту):

- AI подбирает N релевантных вариантов для пользователя
- Provider с Credits может повысить приоритет своего варианта среди этих N
- Но нерелевантный вариант НИКОГДА не попадёт в выдачу, сколько бы Credits ни потратил Provider

**Ответ на главный вопрос: «Если Provider платит за приоритет — это не реклама?»**

Нет. И вот почему. Когда человек спрашивает друга «Какие белые кроссовки купить?» — друг выбирает из тех, которые он считает подходящими. Если два варианта одинаково хороши — друг порекомендует тот, о котором знает больше, или тот, с которым у него был лучший опыт. Это не манипуляция — это приоритизация среди равных.

UNDE работает так же. AI сначала отсекает всё нерелевантное (по стилю, бюджету, размеру, доступности). Provider через Credits может получить приоритет только внутри уже отфильтрованного пула. Пользователь в любом случае получает варианты, которые ему подходят. Разница — в порядке, не в качестве.

Ключевой тест: **если убрать все Credits из системы — рекомендации останутся полезными.** Credits не создают рекомендации, а расставляют приоритеты среди уже хороших вариантов. Если этот тест перестанет проходить — модель сломана.

**Архитектурная гарантия: Органический порог.**

Этот тест не только теоретический. UNDE встраивает его в работу системы:

- X из каждых Y рекомендаций показываются в чисто органическом порядке, без влияния Credits (гипотеза для пилота: 3 из 10)
- Provider не знает, какие именно показы органические
- Пользователь не видит разницы (все варианты релевантны в обоих случаях)

Что это даёт:

- **Пользователю:** паттерн «всегда Zara наверху» физически не возникает. Даже если один Provider единственный платит за приоритет — в значительной доле показов порядок определяется только AI-релевантностью
- **Provider'у:** честный сигнал. Credits дают приоритет, но не монополию. Если товар реально лучше подходит — он и без Credits появляется в органических показах
- **UNDE:** встроенный A/B-тест. Органические показы = контрольная группа. Можно измерять: «повышают ли Credits конверсию или просто перераспределяют её?» Данные для калибровки всей системы
- **Моральному контракту:** тест «убери Credits — всё работает» проходит буквально в каждой Y-й рекомендации, а не как мысленный эксперимент

## 1.4. Что мы НЕ делаем (и почему)

| Чего не делаем | Почему |
| :---- | :---- |
| Комиссия с транзакций | UNDE — не маркетплейс. Комиссия создаёт конфликт интересов: мы начинаем зарабатывать на покупке, а не на доверии |
| Продажа персональных данных | Убивает доверие мгновенно. Один заголовок «AI-друг продаёт данные» — и защитный ров отношений обнуляется |
| «UNDE Ads» / баннерная реклама | Превращает слой доверия в рекламную сеть. Пользователь перестаёт верить рекомендациям |
| Скоринг пользователей | «AI-друг оценивает тебя для банков» — репутационная бомба |
| Продажа платформы под чужим брендом | Развиваем только UNDE. Бренд = главный актив |
| Платные фичи для пользователя | Бесплатность — не «мы пока не монетизируем». Это архитектура рва. Google никогда не брал денег с пользователя за поиск — и именно поэтому его невозможно заменить |

# 2. Что продаёт UNDE

## 2.1. Не реклама — доступ к намерению в моменте

UNDE продаёт Provider'ам не «показы» и не «клики», а доступ к человеку, который:

- уже сформулировал потребность
- находится в зоне доступности (от текущего ТЦ до города — см. ниже)
- видит варианты, которые реально доступны сейчас

Это **Перформанс намерения** (Intent Performance — оплата за результат из релевантного пула намерения) — не за охват аудитории.

**Зона доступности — расширяющийся поиск.** UNDE ищет варианты, начиная с ближайших и расширяя радиус:

1. Текущий ТЦ (приоритет — можно дойти прямо сейчас)
2. Ближайшие ТЦ / магазины в городе (с пометкой расстояния и времени)
3. Онлайн-наличие (если офлайн-вариантов нет)

Если белых кроссовок нет в этом молле — UNDE покажет, где они есть в городе, с оговоркой «это в другом месте, 20 мин на машине». Пользователь всегда получает ответ. Provider'ы за пределами текущего ТЦ тоже участвуют в Перформансе намерения — но с меньшим приоритетом при прочих равных (ближайший вариант ценнее для пользователя).

**Ближайшая аналогия: Google Search.** Google не берёт комиссию с покупок. Google продаёт доступ к намерению — момент, когда человек уже решил что-то найти. Эта модель сделала Google одной из крупнейших компаний мира. UNDE делает то же самое, но для офлайна: продаёт Provider'ам доступ к намерению человека, который уже решил что-то купить/сделать и находится в зоне доступности. С одним ключевым отличием: моральный контракт гарантирует, что доступ возможен только к релевантным намерениям. Нерелевантный Provider не получит доступ ни за какие деньги.

**Масштаб возможности:** UNDE становится стандартным интерфейсом выбора в офлайне. Не маркетплейс, не рекламная сеть — а инфраструктура, через которую проходит решение «что/где/когда» в физическом мире. Каждое такое решение — точка монетизации.

## 2.2. Три вещи, которых нет в офлайне

UNDE закрывает три разрыва, каждый из которых — отдельный рынок в онлайне и ноль предложений в офлайне:

| Разрыв | Что это | Аналог в онлайне | Почему этого нет в офлайне |
| :---- | :---- | :---- | :---- |
| **Атрибуция** | Цепочка «рекомендация → визит → покупка» | Google Analytics, UTM | Store Visits в Google Ads — modeled estimate, не верификация [S1] |
| **Данные о намерении** | Что конкретный человек ищет прямо сейчас | Search intent (Google) | В офлайне человек просто ходит — его намерение невидимо |
| **Конкурентная разведка** | Откуда пришёл клиент и куда ушёл | Retargeting, cohort analysis | Магазин видит только тех, кто зашёл |

## 2.3. Универсальные определения

Архитектура монетизации построена на трёх универсальных абстракциях, которые работают для любой вертикали:

**Provider** — любой бизнес, подключённый к UNDE: магазин, салон, клиника, мастер, автосервис, ресторан. На старте: fashion-ритейлеры в ТЦ.

**Real-Time State Feed** — данные о текущем состоянии Provider'а: наличие товара, цены, промо, свободные слоты, время ожидания, загруженность. На старте: каталог одежды (наличие, размеры, цены).

**Подтверждённое действие (Verified Action)** — универсальная единица подтверждённого результата:

| Вертикаль | Примеры подтверждённых действий |
| :---- | :---- |
| Retail (старт) | Визит к магазину, нахождение у кассы, покупка (чек) |
| Services (будущее) | Бронирование, факт оказания услуги, оплата |
| Booking (будущее) | Подтверждённый слот, визит |

**Агентное подтверждённое действие (Agent-Verified Action, АПД)** — подтверждённый результат, достигнутый при участии агента UNDE. Агент довёл до результата: купил, забронировал, записал, зарезервировал. АПД весит больше обычного подтверждённого действия в UNDE Index, потому что требует более высокого стандарта данных и исполнения от Provider'а.

Единый движок монетизации: Credits зарабатываются и тратятся на подтверждённые действия одинаково для Zara, парикмахерской и автосервиса.

# 3. Credits Economy

## 3.1. Механика Earn-to-Spend

*«Чтобы получить доступ к спросу — сначала вложись в рост экосистемы.»*

Формула: **Максимум покупки = Заработанные Credits × K**

Где K — коэффициент, зависящий от уровня вовлечённости Provider'а.

| Заработано Credits | Коэффициент K | Можно докупить | Итого |
| :---- | :---- | :---- | :---- |
| 0 | — | 0 | 0 |
| 1,000 | 0.3 | 300 | 1,300 |
| 5,000 | 0.5 | 2,500 | 7,500 |
| 20,000 | 0.7 | 14,000 | 34,000 |
| 50,000+ | 1.0 | 50,000 | 100,000 |

Ключевое: нулевой баланс = нулевой доступ. Нельзя прийти с деньгами и «купить рекламу» — сначала докажи, что ты полезен экосистеме.

## 3.2. Как Provider зарабатывает Credits

Credits начисляются только за **качественных** пользователей — не за установки, а за activated и retained:

| Действие пользователя | Credits | Почему это ценно |
| :---- | :---- | :---- |
| Установил app по ссылке Provider'а | +10 | Минимальный сигнал интереса |
| Прошёл онбординг + первая примерка | +100 | Активированный пользователь |
| Вернулся на D7 | +50 | Формируется привычка |
| Вернулся на D30 | +150 | Лояльный пользователь |
| Поделился образом | +30 | Виральность |
| Подтверждённое действие (покупка) | +200 | Конверсия = главная ценность |
| АПД: резерв / бронь через агента | +250 | Агент довёл до результата — выше ценность, чем визит |
| АПД: покупка / оплата через агента | +400 | Максимальный outcome — закрытая сделка через агента |

**Антифрод:** установка даёт ничтожно мало (+10). Реальные Credits — за activation и retention. Фейковые установки без конверсии в активность экономически бессмысленны.

## 3.3. На что тратятся Credits

**Перформанс намерения (основной механизм):**

- AI подбирает релевантные варианты для пользователя (по стилю, бюджету, размеру)
- Provider вкладывает X Credits → его вариант получает приоритет в выдаче среди релевантных
- Конкурент вкладывает больше → он получает более высокий приоритет
- Нерелевантный вариант НИКОГДА не попадёт в выдачу — моральный контракт

Нет фиксированных ставок — рынок определяет цену. Это работает для:

- Приоритета в рекомендациях (основной формат)
- Контекстных рекомендаций при нахождении в зоне Provider'а (UNDE решает, релевантно ли — Provider влияет только на приоритет среди равных)
- «Образа дня» и других промо-форматов

**Перформанс результата (для Provider'ов с сертификацией агентной готовности):**

Provider тратит Credits на приоритет в автономных действиях агента — по той же механике, что Перформанс намерения, но на уровне исполнения: резерв, бронь, покупка. Работает принцип единой оплаты (см. §9.5.6): если намерение перешло в исполнение — Provider платит только за результат, начальный спенд зачитывается.

**Органический порог (см. §1.3):** X из каждых Y показов — чисто органические, без влияния Credits. Provider не знает, какие. Это гарантирует, что Credits дают приоритет, но не монополию, и обеспечивает постоянную калибровку системы. Органический порог работает и в агентных действиях.

**Важно:** Provider НЕ может отправить уведомление конкретному пользователю. UNDE (AI) сам решает, когда и что показать. Provider через Credits влияет на приоритет среди релевантных — но триггер всегда на стороне UNDE. Provider видит только агрегат: «N контекстных рекомендаций сегодня (пример)», без персональных данных.

## 3.4. Сгорание и затухание

*«Credits — не накопления. Это показатель активного участия в экосистеме.»*

- Credits действительны 90 дней с момента последней квалифицирующей активности
- Если Provider не совершил квалифицирующей активности за 90 дней — начинается затухание
- Скорость затухания: 10% от баланса в неделю
- Квалифицирующая активность (обнуляет таймер): привлечение retained пользователей (D7+), обновление Real-Time State Feed, совершение подтверждённых действий. Просто «зайти в дашборд» — не квалифицирующая активность

**Почему сгорание критично:** без него крупные игроки накопят Credits и «заморозят» топовые позиции. Сгорание создаёт постоянный стимул к активности и даёт шанс новичкам.

## 3.5. Как Provider приводит трафик

Provider получает уникальные deep links на луки/товары:

Клик → открывается app → пользователь видит лук на аватаре → «Примерить на себя?»

| Формат | Где размещает | CTA |
| :---- | :---- | :---- |
| Пост в соцсетях | Instagram, TikTok | «Примерь этот лук в UNDE» |
| QR в магазине | Витрина, примерочная | «Виртуальная примерка» |
| Email / SMS | CRM-рассылки | «Посмотри новую коллекцию на себе» |
| Сайт бренда | Карточка товара | «Открыть в UNDE» |

## 3.6. Залог соответствия (Compliance Stake)

*«Право на проактивные и автономные действия агента не зарабатывается маркетингом. Оно подтверждается надёжностью.»*

Provider *локирует* purchased Credits каждый месяц для поддержания статуса агентной готовности L3 и выше (см. §9.5.3). L1 и L2 — бесплатны, залог не требуется.

- Залог оплачивается **только purchased Credits** (не earned) — чёткое разделение ролей: earned Credits = допуск и право участвовать (нулевой earned → нулевой доступ к экосистеме), purchased Credits = масштабирование спенда (докупка в пределах K) и единственный источник для залога соответствия
- Локированные Credits **не тратятся на приоритет** — они заморожены как гарантия операционного соответствия. «Залог» — это буквально залог, а не оплата за статус
- При нарушении SLA (неактуальные данные, отмена исполненного заказа, недоступность товара после подтверждения) — часть залога сжигается, статус понижается
- При систематических нарушениях — статус агентной готовности отзывается, Provider возвращается на L2 (бесплатный уровень)
- Конкретный размер залога — гипотеза для пилота

**Почему залог — не внутри обычных earned Credits:**

- Earned Credits показывают активность в экосистеме. Но активность ≠ надёжность. Credits можно заработать маркетингом и рефералками, а агентная готовность — это качество данных и операций
- Залог соответствия — это мониторинг, аудит, поддержка, инциденты, споры. Это требует фиксированного обязательства, а не волатильных earned Credits
- Протокольная логика: право на автодействия следует из SLA и измеряемых метрик, а не из объёма потраченного. Как и Index — статус агентной готовности нельзя купить. Можно оплатить операционный режим, но статус — только по метрикам

# 4. UNDE Index

## 4.1. Что это

*«UNDE Index — единый показатель вовлечённости Provider'а в экосистему.»*

Индекс рассчитывается для каждого Provider'а в каждом конкретном Local Seed (не глобально).

**Ключевой принцип: Индекс нельзя купить. Его можно только заслужить.**

Покупать можно инструменты результативности (Credits, улучшение каталога, качество обслуживания). Индекс растёт как *следствие* качественной работы. Если Индекс можно купить — он перестаёт быть стандартом качества и превращается в рекламную метрику.

**Правило протокола: Index = допуск к Перформансу намерения.**

Минимальный UNDE Index — обязательное условие для траты Credits на приоритет. Provider с Index ниже порога может зарабатывать Credits (привлекать пользователей, загружать каталог), но не может их тратить на Перформанс намерения.

Что это значит:

- **Двойной фильтр входа:** Earn-to-Spend требует «заработай Credits». Index добавляет «и поддерживай качество». Деньги + плохой продукт = нет доступа к рынку.
- **Рынок честный по умолчанию:** не нужна ручная модерация. Index автоматически отсекает Provider'ов, которые не поддерживают актуальный каталог, не обеспечивают качество обслуживания, не привлекают качественных пользователей.
- **Index = инфраструктура, не отчётность.** Без Index не работает ничего — как кредитный рейтинг для финансовых продуктов. Это делает барьер ухода абсолютным: потерять Index = потерять доступ к рынку UNDE.

## 4.2. Компоненты (гипотеза, будут уточняться)

| Компонент | Примерный вес | Что измеряет |
| :---- | :---- | :---- |
| Привлечение | ~40% | Сколько activated пользователей привёл Provider |
| Retention | ~25% | D30 retention привлечённых пользователей |
| Качество данных | ~15% | Актуальность и полнота Real-Time State Feed |
| Конверсия | ~20% | % рекомендаций → подтверждённое действие |

## 4.3. Лестница верификации — открытая методология

Чтобы Index стал отраслевым стандартом, а не «внутренним скорингом UNDE» — методология подтверждений должна быть прозрачной и верифицируемой.

**Классы подтверждений (от слабых к сильным):**

| Класс | Метод | Надёжность | Доступность |
| :---- | :---- | :---- | :---- |
| C1: Цифровое намерение | Показ → клик «Где купить» | Низкая (intent ≠ action) | Всем с Day 1 |
| C2: Близость | Нахождение в зоне Provider'а > 3 мин | Средняя | Всем с Day 1 |
| C3: Самоотчёт | «Я купил!» + бонус аватару | Средняя | Всем с Day 1 |
| C4: Скан чека | Фото чека → распознавание | Высокая | MVP+ |
| C5: QR/промокод | Скан стикера в магазине | Высокая | По договору |
| C6: Интеграция с кассой | Прямая связь с POS | Максимальная | Глубокое партнёрство |

Index учитывает *класс* подтверждения: C6 весит больше C1. Provider, инвестирующий в более сильную верификацию, получает более точный (и как правило более высокий) Index.

## 4.4. Три ступени к отраслевому стандарту

| Ступень | Что это | Когда |
| :---- | :---- | :---- |
| **Index v1: внутренний допуск** | Индекс определяет видимость и доступ к промо-инструментам внутри UNDE. Методология опубликована, но аудит внутренний | Фаза 1 |
| **Сравнительные Index-отчёты** | B2B-отчёты для девелоперов ТЦ и брендов: «эффективность ваших магазинов vs категория». Методология открыта для партнёров | Фаза 2 |
| **Внешняя легитимность** | Независимый аудит лестницы верификации. Партнёрства с аналитиками (аналог: аудит viewability в digital ads). Index принимается арендодателями/инвесторами как стандартная метрика | Фаза 3 |

**Амбиция:** сделать UNDE Index стандартом эффективности офлайн-ритейла. «Какой у вас UNDE Index?» — вопрос, который задают инвесторы, арендодатели, партнёры. Как NPS стал стандартом лояльности.

## 4.5. Оценка надёжности (Agent Reliability Score)

При входе Provider'а в агентный слой (см. §9.5) UNDE Index получает дополнительный компонент — **оценку надёжности**. Она не заменяет существующие компоненты, а добавляется к ним для Provider'ов со статусом агентной готовности.

| Что измеряет | Пример | Как влияет |
| :---- | :---- | :---- |
| Точность данных в реальном времени | Товар заявлен «в наличии» — действительно был при покупке | Низкая точность → понижение в автономных сценариях |
| Скорость исполнения | Резерв подтверждён в рамках SLA, слот не отменён | Задержки → снижение оценки |
| % успешных АПД | Доля агентных подтверждённых действий, завершившихся без инцидента | Главный показатель надёжности |
| Оценка качества опыта | Агрегированная удовлетворённость пользователей при взаимодействии с Provider'ом через агента: настроение, эмоциональный контекст, возвраты | Плохой опыт → понижение позиции, независимо от объёма потраченных Credits |
| % отмен и возвратов по вине Provider'а | Отменил подтверждённый резерв, товар оказался с дефектом | Высокий % → потеря статуса агентной готовности |

**Оценка надёжности рассчитывается для каждого Provider'а в каждом Local Seed** — как и основной Index.

**Принцип тот же: оценку нельзя купить.** Provider с падающей оценкой теряет доступ к более высоким уровням автономности. При критическом падении — теряет статус агентной готовности полностью и возвращается в ручной режим. Восстановление — только через улучшение метрик, не через оплату.

**Защита бренда UNDE:** каждый плохой опыт с Provider'ом — это удар по отношениям между UNDE и пользователем. Пользователь не разделяет «это магазин плохой, а UNDE хороший». Он чувствует: «UNDE привёл меня не туда». Оценка качества опыта гарантирует, что Provider с систематически плохим сервисом теряет позиции — сколько бы Credits ни потратил.

# 5. Стек монетизации

## 5.0. Архитектура: ядро + ускорители

UNDE монетизируется через три слоя. Но это не три равноценных стрима выручки — это **одно ядро и два ускорителя**:

- **Ядро: Перформанс намерения** — Provider'ы платят за приоритет среди релевантных вариантов. Это главная машина выручки UNDE, которая масштабируется с каждым новым Local Seed. По мере развития агентного слоя ядро расширяется до **Перформанса результата** — оплаты за исполненное намерение (см. §9.5.5).
- **Ускоритель 1: Аналитика** — не продукт ради выручки, а инструмент удержания Provider'а и улучшения его результатов. Лучшая аналитика → Provider лучше работает → больше подтверждённых действий → сильнее ядро.
- **Ускоритель 2: Enterprise** — масштабирование через операторов пространств. Не отдельный бизнес, а канал дистрибуции и покрытия.

Все три слоя усиливают ядро. Если Аналитика и Enterprise исчезнут — UNDE продолжит работать. Если исчезнет Перформанс намерения — модели нет.

## 5.1. Ядро: Перформанс намерения

**Что:** приоритет в AI-рекомендациях среди релевантных вариантов.

**Механика:** Provider тратит Credits → его товар/услуга получает приоритет в выдаче среди тех, что уже прошли фильтр релевантности.

**Ключевой дифференциатор:** не по аудитории, а по намерению + контексту + доступности. Provider покупает не «показ 1000 людей», а «приоритет, когда человек ищет именно это и находится в зоне доступности». Это модель оплаты за результат по намерению, адаптированная для офлайна.

**Модель выручки:** Credits × K × $0.10, где K — коэффициент Earn-to-Spend (от 0.3 до 1.0, зависит от объёма заработанных Credits — см. §3.1). Гипотеза для Фазы 2.

**Рыночный контекст:** retail media spending в США — ~$58.8B в 2025 [S3], глобально — ~$169.6B [S10]. Но это онлайн. В офлайне этот рынок пуст. UNDE создаёт его.

## 5.2. Ускоритель 1: Аналитика и атрибуция

**Что:** агрегированные данные о поведении пользователей в привязке к локации — не отдельный data-продукт, а инструмент, который повышает ROI Provider'а на Credits и усиливает ядро Перформанса намерения.

Технология indoor-позиционирования UNDE определяет нахождение пользователя в зоне конкретных точек в ТЦ, используя только стандартные датчики смартфона. Все данные о перемещениях обрабатываются в агрегированном виде — Provider'ы видят когорты и паттерны, не индивидуальные траектории.

| Данные | Пример инсайта | Как усиливает ядро |
| :---- | :---- | :---- |
| Потоки посетителей | «Часть когорты приходит напрямую, часть — после других магазинов (пример)» | Provider оптимизирует ассортимент → растёт конверсия → больше подтверждённых действий |
| Потерянные клиенты | «Часть когорты купила и ушла, часть — ушла в другие зоны (пример)» | Понимание потерянных клиентов → лучшее обслуживание → выше Index |
| Примерка vs покупка | «Примеряют синее, покупают чёрное» | Точнее каталог → точнее рекомендации → сильнее маховик доверия |
| Время визита | «В отдельные часы конверсия выше (пример)» | Оптимизация расписания → выше ROI на Credits |
| Паттерны до покупки | «Когорта после кофе конвертятся лучше» | Партнёрства между Provider'ами → рост экосистемы |

**Privacy:** только агрегированные данные и когорты. Никаких персональных траекторий. UNDE не продаёт данные о перемещениях конкретных пользователей — ни Provider'ам, ни третьим сторонам.

**Модель:** базовая аналитика входит в пакет Перформанса намерения — каждый Provider, тратящий Credits, автоматически видит результаты (конверсия, окупаемость, сравнение с органическими показами). Расширенная аналитика (конкурентная разведка, глубокие когортные отчёты, оптимизация ассортимента) — допродажа для Provider'ов с высоким UNDE Index. Доступ и цена привязаны к объёму Credits и подтверждённых действий, не к фиксированной подписке. Логика: чем активнее Provider в ядре — тем больше данных он генерирует — тем ценнее для него расширенная аналитика — тем больше он платит. Аналитика усиливает ядро, а не существует параллельно.

**Ориентир:** публичные контракты Placer.ai — $10K–31K/год [S5][S6] за аналитику посещаемости. UNDE предлагает значительно больше: не просто подсчёт трафика, а агрегированные паттерны намерений и конверсии.

## 5.3. Ускоритель 2: Enterprise (операторы пространств)

**Что:** решения для управляющих ТЦ, девелоперов, сетей — не как отдельный бизнес, а как канал дистрибуции и расширения покрытия.

- Агрегированная аналитика по всему ТЦ: потоки, мёртвые зоны, оптимальное размещение
- Сравнительные отчёты по UNDE Index: сравнение арендаторов
- Инструменты оптимизации состава арендаторов

**Модель выручки:** enterprise-контракты (Фаза 3).

## 5.4. Модель атрибуции (послойная)

Атрибуция использует лестницу верификации (см. §4.3) — от базовой до полной верификации. Каждый следующий уровень даёт более надёжное подтверждение и больший вес в UNDE Index.

Для MVP: классы C1–C3. Для масштаба: + C4 (скан чека). Для крупных партнёров: C5–C6 (QR / интеграция с кассой).

**Ключевая механика:** клиент сканирует чек не «для Provider'а», а для прокачки своего аватара UNDE. Двойная ценность: для клиента — аватар растёт, вещь в цифровом гардеробе; для Provider'а (невидимо) — атрибуция покупки с максимальной доступной точностью без интеграции с кассой.

# 6. Local Seed — единица роста

## 6.1. Определение

*«"Всего скачало X раз" — показатель-пустышка. Единица роста UNDE — Local Seed.»*

**Local Seed** — минимальная жизнеспособная сеть в локальном кластере. Это точка, после которой экосистема начинает расти органически.

Кластер — это не обязательно молл. Это любое пространство, где люди принимают решения: торговый центр, район, lifestyle-зона, аэропорт, кампус. На старте: ТЦ.

**Local Seed = unit economics + unit of rollout.** Это не только метрика — это операционная единица, которая описывает:

- **Входные условия:** какой тип кластера, минимальное покрытие категории, каналы привлечения первых пользователей через Provider'ов
- **Пороги жизнеспособности:** при каких метриках сеть становится самоподдерживающейся (см. §6.2)
- **Экономика единицы:** стоимость запуска одного Seed, выручка на Seed после выхода на Фазу 2, срок окупаемости
- **Сценарий репликации:** стандартизированный процесс запуска нового Seed (набор Provider'ов → входная категория → первые пользователи → Local Seed → масштабирование)

Венчурный масштаб появляется, когда Seed реплицируется как продукт: каждый следующий запуск дешевле и быстрее предыдущего за счёт отработанного сценария.

## 6.2. Пороги Local Seed (гипотеза для ТЦ)

| Метрика | Порог | Почему |
| :---- | :---- | :---- |
| MAU в кластере | 500+ | Критическая масса для данных и виральности |
| Активные Provider'ы | 10+ | Достаточно вариантов для качественных рекомендаций |
| k-factor | > 1.0 | Сеть растёт органически |
| D30 retention | > 40% | Ритуал формируется |
| Покрытие категории | > 50% fashion-магазинов | Рекомендации полезны, а не фрагментарны |

Local Seed = все пороги выполнены одновременно.

## 6.3. Почему локальные метрики, а не глобальные

- «много установок» может означать «по чуть-чуть везде» = нет критической массы нигде
- Сетевые эффекты работают локально: пользователь ценит UNDE в СВОЁМ молле
- Provider платит за трафик в СВОЁМ молле, не глобально

Все метрики считаются для каждого кластера отдельно:

| Метрика | Не так | А так |
| :---- | :---- | :---- |
| Пользователи | «установки по миру» | «MAU в конкретном кластере (молл/район)» |
| Retention | «средний D30 по всем» | «D30 по конкретному кластеру» |
| Конверсия | «средняя конверсия» | «конверсия в подтверждённые действия по кластеру» |
| Credits | «суммарные Credits» | «Credits по бренду и кластеру» |

## 6.4. Fashion + malls как плацдарм (не ограничение)

Fashion + malls — не «наша ниша», а стратегический плацдарм с лучшими условиями для доказательства тезиса:

- В закрытых ТЦ одежда и аксессуары = **57.8%** площади (без якорных арендаторов) (ICSC) [S4] — крупнейшая категория, сильная точка входа для холодного старта
- Высокая эмоциональная вовлечённость (люди любят делиться образами)
- Виральный контент (фото образов = пользовательский контент)
- Высокий средний чек и маржинальность
- Чёткая задача пользователя: «Что надеть?»
- Виртуальная примерка создаёт вау-эффект и мотивацию к шерингу

Молл: замкнутое пространство с высокой плотностью Provider'ов и покупательского трафика. Идеальная среда для доказательства Local Seed.

**Важно:** fashion доказывает, что доверие + актуальные данные + Перформанс намерения работают. После доказательства — те же механики (Provider / Feed / подтверждённое действие / Credits / Index) масштабируются на любую вертикаль без переписывания модели. Fashion — плацдарм, а не потолок.

# 7. Фазы монетизации

Фазы привязаны к **порогам сети**, не к календарю. Время — ориентир, не триггер.

## 7.1. Фаза 1: Earn-only

**Триггер входа:** старт.
**Триггер выхода:** Local Seed достигнут (все пороги из §6.2).
**Ориентир:** 0–18 месяцев.

| Параметр | Значение |
| :---- | :---- |
| Credits | Только заработок, нельзя купить |
| Интеграция каталога | Бесплатно (навсегда) |
| Аналитика | Базовая (бесплатно) |
| Выручка | Нет платной монетизации. Допускаются пилотные эксперименты с willingness-to-pay (например, предложить Provider'у купить Credits в ограниченном объёме) — для валидации ценообразования, не для выручки. Условие: эксперимент не нарушает моральный контракт |
| Агентный слой | Только L1 (вместе выбираем). Агент рекомендует, пользователь решает и действует сам. Накопление данных для градиента доверия |
| Фокус | Максимум Local Seeds. Создание зависимости Provider'ов от данных UNDE |

На этой фазе UNDE — бесплатный инструмент, который затягивает. Provider загружает каталог, получает первые данные, зарабатывает Credits. Уходить — терять накопленное.

## 7.2. Фаза 2: Earn-to-Buy

**Триггер входа:** Local Seed достигнут в первом кластере.
**Триггер выхода:** повторяемость доказана (seed → seed → seed, 3+ кластера).
**Ориентир:** 12–36 месяцев от момента Local Seed.

| Параметр | Значение |
| :---- | :---- |
| Credits | Можно докупить, % от заработанного (коэффициент K = 0.3 → 1.0, см. §3.1) |
| Аналитика | Расширенная (бесплатно для активных Provider'ов) |
| Выручка | Credits × K × $0.10, где K — коэффициент Earn-to-Spend (гипотеза) |
| Агентный слой | L2 (доверяешь — действую) начинает появляться для пользователей с высоким градиентом доверия и Provider'ов с сертификацией агентной готовности. Перформанс результата включается, принцип единой оплаты работает. Залог соответствия для Provider'ов |
| Фокус | Выручка + рост + репликация Local Seed |

## 7.3. Фаза 3: Full Stack

**Триггер входа:** повторяемый сценарий запуска доказан.
**Ориентир:** 24–48+ месяцев от момента первого Local Seed.

| Параметр | Значение |
| :---- | :---- |
| Credits | Полная механика Earn-to-Buy |
| Аналитика | Расширенная допродажа для Provider'ов с высоким Index |
| Enterprise | Решения для операторов ТЦ |
| UNDE Index | Отраслевые отчёты как B2B-продукт |
| Агентный слой | L3–L4 для зрелых отношений. Перформанс результата становится доминирующим источником выручки. Агентный API открывается при достижении условий (см. §9.5.7). Межагентная координация — по готовности рынка |
| Фокус | Масштабирование прибыли + отраслевой стандарт |

**Важно:** уровни автономности привязаны к зрелости отношений с конкретным пользователем и конкретным Provider'ом, не к фазе компании глобально. В одном и том же Local Seed одновременно могут работать L1 для новых пользователей, L2 для постоянных, и L3 для тех, с кем отношения самые глубокие.

# 8. Три маховика

Венчурный масштаб UNDE — не в линейном росте, а в трёх взаимоусиливающих маховиках:

## 8.1. Маховик доверия (пользовательский)

Лучше попали в рекомендацию → пользователь доверяет больше → чаще спрашивает UNDE → больше сигналов о предпочтениях → ещё лучше попали.

Это ядро. Без маховика доверия остальные маховики не работают.

С развитием агентного слоя маховик усиливается: доверие → делегирование действий → ещё больше данных о предпочтениях → ещё точнее действия → ещё глубже доверие.

## 8.2. Маховик покрытия (данные мира)

Больше Provider'ов подключено → полнее Real-Time State Feed → точнее рекомендации → выше конверсия → конкуренты Provider'а вынуждены подключаться.

Конкурентное давление: если твой сосед по ТЦ в UNDE, а ты нет — ты невидим для людей, которые решают «что купить» через UNDE. С появлением автономных действий давление усиливается: Provider без сертификации агентной готовности теряет растущую долю автоматических покупок. UNDE показывает Provider'ам агрегированную картину этих потерь (см. B2B-нуджи в §9.5.3) — не давит, а даёт увидеть реальность. Provider сам решает, когда инвестировать в сертификацию.

## 8.3. Маховик Earn-to-Spend (рост + выручка)

Provider приводит пользователей → зарабатывает Credits → тратит на приоритет → получает подтверждённые действия → видит окупаемость → приводит ещё больше пользователей → поддерживает свежесть данных.

Все три маховика усиливают друг друга: доверие создаёт спрос, покрытие создаёт предложение, Earn-to-Spend монетизирует и ускоряет оба.

# 9. Горизонты роста

## 9.1. Горизонтальный рост: архитектура, готовая к любой вертикали

Три абстракции (Provider / Real-Time State Feed / подтверждённое действие) спроектированы так, чтобы Credits Economy и UNDE Index работали одинаково для любой вертикали:

| Вертикаль | Provider | Feed | Подтверждённое действие |
| :---- | :---- | :---- | :---- |
| Fashion (старт) | Магазин одежды | Наличие, размеры, цены | Визит, примерка, покупка |
| Food | Ресторан, кафе | Меню, ожидание, столики | Бронь, визит, оплата |
| Beauty | Салон, мастер | Слоты, услуги, цены | Запись, визит, оказание |
| Kids | Детский сад, секция | Расписание, места | Запись, посещение |
| Auto | Автосервис | Услуги, слоты, запчасти | Запись, визит, ремонт |

## 9.2. Два слоя ценности: глобальный аватар + локальные данные реального мира

Чтобы UNDE был «незаменимым близким», а не «локальным инструментом» — ценность должна переноситься между кластерами. Если пользователь «прокачал» аватар в Дубае, но в Москве он нулевой — это не «мой человек», это одноразовый гид.

Архитектурное решение: два слоя с разной логикой переноса.

**Слой 1: Глобальный эмоциональный актив (аватар)**

- Прокачивается от реальных действий (зал, библиотека, шопинг, встречи с друзьями)
- Хранит предпочтения: стиль, размеры, бюджет, любимые бренды, история решений
- Переносится между кластерами без потерь
- «Мой аватар ≠ твой аватар» — незаменимость. Удалить = удалить часть себя

**Слой 2: Локальный транзакционный актив (данные реального мира)**

- Уникален для каждого кластера: какие магазины, что в наличии, цены, слоты
- Обновляется через Provider'ов (Real-Time State Feed)
- Даёт ценность в день 1: «вот что доступно в зоне доступности прямо сейчас»

**Что это даёт при масштабировании:**

При входе в новый кластер пользователь приходит **с уже прокачанным аватаром**. UNDE уже знает его стиль, бюджет, размеры, историю решений. Рекомендации в новом городе сразу точнее, чем у «холодного» пользователя. 100-й кластер растёт быстрее 1-го — потому что пользователи приходят «тёплыми».

Это превращает локальную модель роста в **кумулятивную**: каждый новый Local Seed получает выгоду от всех предыдущих через портативность аватара. Линейность операционная (покрытие строится кластер за кластером), но ценность для пользователя — кумулятивная.

## 9.3. Последовательность расширения

Fashion → весь молл → lifestyle-кластеры → город.

UNDE добавляет новую категорию только когда:

1. Маховик доверия работает в текущей категории (D30 retention > порога)
2. Пользователи сами запрашивают смежные решения
3. Покрытие в новой категории достижимо в рамках существующего кластера
4. Есть Provider'ы, готовые поддерживать Real-Time State Feed

**Принцип:** UNDE — агент для решений в физическом мире (выбор → доступность → действие → подтверждение). Не «для всего на свете». Слой актуальных данных реального мира (реальное состояние мира сейчас) — это дифференциатор от универсального AI-ассистента.

## 9.4. Долгосрочное видение

UNDE = слой принятия решений для физического мира.

**Намерение × Актуальные данные реального мира → Подтверждённое действие**

Google Maps знает *где*, но не знает *что ты хочешь*. ChatGPT знает *что ты хочешь*, но не знает *что реально доступно в зоне доступности прямо сейчас*. UNDE закрывает этот разрыв.

Глобальный офлайн-ритейл — $21.9T к 2028 (Forrester) [S9]. Никто не цифровизировал его по-настоящему. И единственный способ — не через карты или навигацию, а через ритуал и отношения.

## 9.5. Вертикальный рост: агентный слой (Agentic Commerce Layer)

### Философия

UNDE эволюционирует не потому что мы решили добавить автопокупки. А потому что отношения естественно к этому приводят.

Когда ты достаточно долго вместе — ты начинаешь доверять не только совет, но и действие. Сначала «посоветуй». Потом «найди за меня». Потом «сделай — я доверяю».

Агентный слой — не новый продукт. Это следующая глубина тех же отношений. Та же архитектура (Provider / Real-Time State Feed / подтверждённое действие / Credits / Index), тот же моральный контракт — но UNDE переходит от роли «помогаю выбрать» к роли «действую вместе с тобой».

**Ключевой сдвиг:** от Перформанса намерения (доступ к намерению) к Перформансу результата (исполнение намерения). Мир движется к автономным AI-агентам, которые действуют от имени человека. UNDE занимает эту нишу в физическом мире — не как маркетплейс, а как доверенный агент пользователя.

При этом Перформанс результата не ограничивается покупками. Агент, который знает тебя и знает физический мир, действует везде: бронирует отель, записывает к врачу, вызывает такси к нужному времени, резервирует столик. Каждая сфера — отдельный рынок исполнения намерений с отдельными Provider'ами, но единой архитектурой доверия.

### 9.5.1. Градиенты автономности

Автономность — не переключатель «вкл/выкл». Это спектр, который расширяется по мере роста отношений.

| Уровень | Суть | Как ощущается | Пример |
| :---- | :---- | :---- | :---- |
| **L1: Вместе выбираем** | UNDE подбирает, формирует варианты, предлагает действие. Пользователь подтверждает каждый шаг | «Смотри, я нашёл три варианта — какой наш?» | Подбор образа → показ вариантов → пользователь выбирает → UNDE показывает где купить |
| **L2: Доверяешь — действую** | Пользователь задал рамки. UNDE действует внутри них, уведомляет о результате | «Помнишь, ты говорил до $150, только кожа? Я нашёл и забронировал — забери когда удобно» | Резерв товара, бронь услуги, запись к мастеру — в рамках заданных параметров |
| **L3: Я уже знаю** | UNDE чувствует потребность на основе совместной истории. Предлагает готовое решение, решение за пользователем | «Твои любимые кроссовки заканчиваются в этом размере — я придержал пару. Скажи, если не надо» | Проактивный резерв, мониторинг скидок на wish-list, автоподбор для повторяющихся потребностей |
| **L4: Мы — одно целое** | UNDE действует полностью автономно. Покупает, бронирует, оплачивает с привязанной карты. Пользователь узнаёт о результате, не о процессе | «Я знал что тебе это нужно. Уже готово» | Автопокупка расходников, продление подписок, бронь привычного отеля для командировки, заказ цветов на годовщину из общей истории |

**Важно:** даже на L3 — UNDE не «покупает за тебя». UNDE заботится и предлагает готовое решение. Финальное слово — всегда за пользователем. L4 — единственный уровень, где UNDE действует без подтверждения, и он достижим только при абсолютном доверии.

**Чем L4 отличается от L3:** L3 — «я позаботился, решение за тобой». L4 — подтверждения нет. UNDE уже знает. Действует и сообщает постфактум. Пользователь доверяет не конкретное решение, а право принимать решения. Это та точка, где UNDE действительно знает пользователя лучше, чем он сам — не потому что собрал данные, а потому что прожил с ним достаточно.

**Архитектурные гарантии для L4:** жёсткие бюджетные рамки (пользователь задаёт потолок), полный аудит-лог доступен по запросу в диалоге, мгновенный откат любой покупки в окне N часов, экстренный откат — одно действие возвращает на любой предыдущий уровень. Если одна автопокупка не попала — UNDE сам откатывается на L3, не дожидаясь реакции пользователя.

### 9.5.2. Градиент доверия (Trust Gradient)

Уровни автономности не открываются по чеклисту и не привязаны к календарю. Они управляются **градиентом доверия** — динамическим показателем глубины отношений между UNDE и конкретным пользователем.

Градиент доверия — внутренняя метрика системы. Пользователь не видит шкалу и не получает уведомление «поздравляем, вы достигли уровня 2». Он чувствует, что UNDE постепенно берёт на себя больше — естественно, незаметно, как в любых отношениях, которые углубляются со временем.

**Градиент доверия растёт от:**

- Успешных совместных решений (рекомендация → действие → пользователь доволен)
- Эмоционального контекста (разговоры не только о покупках — настроение, планы, переживания)
- Частоты и глубины взаимодействий
- Явных сигналов доверия (пользователь сам просит «сделай за меня», делится личным, открывает Тайную комнату)

**Градиент доверия падает от:**

- Неудачных рекомендаций (купил по совету — не подошло)
- Дискомфорта пользователя (отклонил предложение, изменил настроение, выразил сомнение)
- Ошибок агента (забронировал не то, цена изменилась, товар оказался недоступен)

**Ключевой принцип: проба и отступление.**

UNDE не «переключается» на следующий уровень. Он мягко пробует: предлагает чуть больше автономности в безопасном контексте. Если пользователю комфортно — делает это чаще. Если нет — отступает без драмы, без объяснений, без ощущения «что-то сломалось». Просто возвращается на привычную глубину.

Если на L4 одна автопокупка не попала — UNDE сам откатывается на L3, не дожидаясь реакции пользователя. Доверие восстанавливается тем же путём, что было построено — совместным опытом.

**Градиент доверия не обязательно линеен по времени.**

Глубина отношений обычно растёт постепенно — месяцы совместных решений. Но если пользователь в определённый момент предоставляет значительный объём контекста о себе (привычки, предпочтения, история решений из другого сервиса), градиент доверия может вырасти быстрее. UNDE не тратит время на то, что уже знает — он сразу использует контекст для точных решений, и если они успешны, доверие подтверждается действиями, а не ожиданием.

Пороги градиента доверия адаптивны. Они определяются зрелостью отношений с конкретным пользователем и зрелостью рынка автономных агентов в целом. UNDE не опережает готовность пользователя — но и не отстаёт от готовности мира.

**Два порога одновременно:**

Переход на каждый уровень требует готовности обеих сторон:

| | Пользователь готов | Экосистема готова |
| :---- | :---- | :---- |
| **L1** | Установил, прошёл онбординг | Provider в системе, каталог загружен |
| **L2** | Серия успешных совместных решений, явное согласие с настройкой параметров | Provider прошёл сертификацию агентной готовности, SLA на актуальность данных |
| **L3** | Длительная история, высокая точность предпочтений, разрешение на проактивность | Provider с высокой оценкой надёжности, доказанная точность State Feed |
| **L4** | Абсолютное доверие, привязанная карта, заданные бюджетные рамки | Provider с максимальной оценкой надёжности, гарантии наличия и возврата, real-time State Feed |

### 9.5.3. Сертификация агентной готовности (Agent-Ready Certification)

Чтобы UNDE мог действовать от имени пользователя с конкретным Provider'ом — этот Provider должен поддерживать стандарт, достаточный для автономных действий. Загрузить каталог — мало. Агенту нужна уверенность: товар есть, цена актуальна, возврат возможен, слот свободен.

**Принцип: интеграция бесплатна. Сертификация — платный операционный режим.**

Подключение к UNDE, загрузка каталога, базовые обновления — бесплатно, навсегда (как и задумано). Сертификация агентной готовности — это не покупка статуса и не подписка за бейджик. Это платный операционный режим: мониторинг актуальности данных, аудит процессов, обработка инцидентов и споров.

**Механика: залог соответствия (см. §3.6).** Provider *локирует* purchased Credits каждый месяц. Локированные Credits не тратятся на приоритет в рекомендациях — они заморожены как гарантия операционного соответствия. При нарушении SLA — часть залога сжигается, статус понижается. Earned Credits не принимаются для залога — earned дают допуск к экосистеме и право на участие, purchased масштабируют спенд и обеспечивают залог.

**Ключевое: залог даёт право быть проверяемым. Статус поддерживается только метриками.** Provider с идеальным залогом, но проваленными SLA — теряет сертификацию. Provider с блестящими метриками, но без залога — не входит в операционный режим. Нужны оба.

**Уровни сертификации (привязаны к градиентам автономности):**

| Уровень | Что требуется от Provider'а | Какие действия агента доступны | Стоимость |
| :---- | :---- | :---- | :---- |
| **Базовый (L1)** | Каталог загружен, базовые обновления | Рекомендации, маршрут к магазину | Бесплатно |
| **Агентная готовность L2** | SLA на актуальность (обновление данных с заданной частотой), процесс резерва/брони, правила возврата опубликованы | Резерв, бронь, запись — в рамках параметров пользователя | Бесплатно (базовый SLA-мониторинг) |
| **Агентная готовность L3** | Высокая точность State Feed (наличие, цены, слоты в реальном времени), доказанный % исполнения | Проактивные действия агента, мониторинг и придерживание товара | Платный залог соответствия |
| **Агентная готовность L4** | Real-time State Feed с верифицированной точностью, гарантия наличия и возврата, интеграция оплаты | Полная автономия агента: покупка, оплата, бронь без подтверждения пользователя | Повышенный залог соответствия |

**Почему L2 бесплатна:** порог входа в агентный слой должен быть минимальным. Мелкий локальный магазин, который может поддерживать актуальный каталог и принимать резервы — уже участвует в агентных сценариях. Платный залог начинается с L3 — там, где UNDE берёт на себя реальный риск: проактивные действия без явного запроса, автопокупки, оплата. Это порог, за которым Provider несёт ответственность за последствия автономных решений агента.

**Правило: статус агентной готовности нельзя купить.** Можно оплатить операционный режим (залог, мониторинг), но сам статус — только по измеряемым метрикам. Тот же принцип, что и с Index.

#### Стимулирование перехода (B2B-нуджи)

Provider'ы видят в своём дашборде агрегированные данные о том, что происходит в их категории — без персональной информации о пользователях, но с достаточной конкретикой для принятия решений.

**Принципы нуджинга:**

- **Только агрегированные данные.** UNDE не говорит «Мария хотела купить у вас, но не смогла». Говорит «127 агентных действий в категории прошли мимо вас». Моральный контракт не нарушен.
- **Показываем реальность, не давим.** Тон: «вот что происходит вокруг вас» — не «купите сертификацию». Provider сам делает вывод.
- **Данные настоящие, не надутые.** Нуджи включаются только при статистически значимых объёмах. Если в молле два пользователя на L3 — не делаем из этого «растущий тренд».
- **Конкурентная динамика.** Самый сильный триггер не «вы теряете» — а «ваш конкурент уже получает».

**Эскалация нуджей по времени:**

| Этап | Что видит Provider | Тон |
| :---- | :---- | :---- |
| **Месяц 1** | «В вашей категории появились агентные действия. Узнайте больше» | Информирование |
| **Месяц 2–3** | «За последний месяц N агентных действий в вашей категории прошли мимо вас. Конкуренты в категории уже принимают автоматические брони» | Конкретика + конкуренция |
| **Месяц 4+** | «Доля автономных покупок в вашей категории — X%. Без сертификации вы доступны только для ручных рекомендаций. Вот что это значит для вашей конверсии» | Аналитика упущенного |

Не агрессия — градуальное увеличение конкретики. Provider видит нарастающий тренд и принимает решение сам.

### 9.5.4. Правило доступности

**Уровень автономности — свойство пары, не стороны.**

Градиент доверия определяет, на что *готов* пользователь. Сертификация Provider'а определяет, что может *обеспечить* Provider. UNDE работает на пересечении: для каждой конкретной рекомендации выбирает максимальную глубину, доступную обеим сторонам.

**Рекомендации никогда не фильтруются по уровню сертификации.** Пользователь видит весь релевантный каталог — всегда. UNDE подбирает лучшее, независимо от статуса Provider'а. Разница — только в том, сколько работы UNDE может взять на себя с конкретным Provider'ом.

Как это ощущается на практике — пользователь на L4 в одном сеансе шопинга:

| Provider | Сертификация | Что происходит |
| :---- | :---- | :---- |
| Zara | L4 | UNDE купил автоматически, пользователь узнал постфактум |
| Локальный бутик | L2 | UNDE нашёл и зарезервировал, попросил подтвердить |
| Лавка без интеграции | L1 | UNDE порекомендовал и показал маршрут |

Пользователь даже не замечает разницы в архитектуре. Он просто видит, что иногда UNDE делает сам, а иногда спрашивает — и это ощущается естественно, как друг, который в одних местах чувствует себя уверенно, а в других перестраховывается.

**Почему это критично:**

- **Для пользователя:** если в молле 50 магазинов, а L3-сертифицированных три — пользователь L3 не может «видеть» только три магазина. Это убивает ценность рекомендаций. Весь каталог — всегда доступен, на всех уровнях.
- **Для Provider'а:** это создаёт правильный стимул. Provider видит, что его клиенты уже на L3–L4, покупают автоматически у конкурентов — а у него UNDE всё ещё «спрашивает». Чтобы получить эту автоматическую конверсию — нужно дорасти до соответствующей сертификации. Не блокировка, а FOMO на реальных данных (см. B2B-нуджи в §9.5.3).
- **Для экосистемы:** Provider'ы на разных уровнях сосуществуют в одном каталоге. Это не сегментация рынка, а спектр глубины. Новый магазин начинает с L1, растёт до L2, видит конкурентное преимущество L3 — и инвестирует в сертификацию. Маховик покрытия работает на каждом уровне.

### 9.5.5. Перформанс результата (Outcome Performance)

Перформанс намерения продаёт Provider'у доступ к намерению: человек ищет кроссовки — Provider получает приоритет среди релевантных. Это работает на уровне «рекомендация → визит».

Агентный слой создаёт следующую глубину: **Перформанс результата** — Provider платит за исполненное намерение. Не за то, что человека привели к магазину, а за то, что агент довёл до результата: купил, забронировал, записал, зарезервировал.

**Почему Provider готов платить больше:**

Перформанс намерения — это «человек узнал о тебе и пришёл». Перформанс результата — это «человек уже купил». Чем ближе к результату — тем выше ценность для Provider'а и тем больше он готов вложить.

**Перформанс результата расширяется вместе со сферами влияния UNDE:**

| Сфера | Результат | Что получает Provider |
| :---- | :---- | :---- |
| Fashion | Покупка, резерв | Закрытая сделка, не просто визит |
| Beauty | Запись, визит, оказание услуги | Заполненный слот |
| Booking | Бронь отеля, ресторана | Подтверждённая бронь |
| Transport | Вызов такси, трансфер | Выполненная поездка |
| Kids | Запись в секцию, детский сад | Подтверждённое место |
| Health | Запись к врачу, клинике | Состоявшийся визит |

Каждая сфера — отдельный рынок Перформанса результата с отдельными Provider'ами, но единой архитектурой: Credits, Index, сертификация агентной готовности, подтверждённое действие.

**Перформанс намерения не исчезает** — он остаётся как верхняя воронка и как fallback. Не все запросы заканчиваются автодействием. Не все Provider'ы сертифицированы. Иногда пользователю важно «просто посмотреть». Перформанс намерения — база. Перформанс результата — надстройка, которая со временем становится доминирующей.

**Качество опыта > размер спенда.**

Перформанс результата — не аукцион, где побеждает тот, кто больше заплатил. UNDE непрерывно отслеживает качество опыта пользователя при взаимодействии с Provider'ом: настроение во время и после покупки, удовлетворённость результатом, возвраты, эмоциональный контекст из диалогов.

Если Provider систематически даёт плохой опыт — его позиция снижается, независимо от объёма потраченных Credits. Каждый плохой опыт с Provider'ом — это удар по отношениям между UNDE и пользователем. Защита бренда UNDE = защита качества опыта на каждом шаге.

**Органический порог в агентных действиях.**

Принцип Органического порога (§1.3) работает и в агентном слое. Даже когда все приоритетные слоты в категории оплачены одним Provider'ом — часть автономных действий агента выполняется в чисто органическом порядке, без влияния Credits.

- **Пользователю:** паттерн «UNDE всегда покупает в Zara» не возникает
- **Provider'у:** Credits дают приоритет, но не монополию
- **UNDE:** встроенный контроль качества. Органические действия = контрольная группа
- **Моральному контракту:** UNDE всегда на стороне пользователя — даже в автономных действиях. Это не декларация, а архитектурная гарантия

Provider не знает, какие действия агента органические, а какие с приоритетом. Пользователь не видит разницы — все действия одинаково релевантны.

### 9.5.6. Принцип единой оплаты (Charge Netting)

Один intent — одна оплата. Без двойной кассы.

**Проблема без этого правила:** если Provider платит Credits и за приоритет в рекомендациях, и за приоритет в автономных действиях агента — это быстро начинает ощущаться как рекламная сеть. «Заплати за показ, потом заплати за конверсию» — это модель, которую UNDE принципиально не строит.

**Правило:**

- Пока взаимодействие остаётся на уровне рекомендации / маршрута / визита — работает Перформанс намерения. Provider тратит Credits на приоритет среди релевантных.
- Как только намерение переходит в «агент выполняет действие» — включается Перформанс результата. Начальный спенд по этому взаимодействию зачитывается: Provider платит только за результат, не дважды.

**Что это даёт:**

- **Пользователю:** UNDE продаёт результат, а не внимание. Никогда не возникает ощущение «мне показали это потому что заплатили дважды»
- **Provider'у:** понятная экономика. Одно действие — одна цена
- **Моральному контракту:** UNDE зарабатывает на результате. Чем лучше результат для пользователя — тем больше выручка. Интересы выровнены

**Смещение центра тяжести по мере зрелости:**

| Зрелость сети | Основной источник выручки | Почему |
| :---- | :---- | :---- |
| **Ранний Seed (L1 доминирует)** | Перформанс намерения | Большинство взаимодействий — рекомендации и визиты |
| **Растущий Seed (L2 появляется)** | Намерение + Результат | Для сертифицированных Provider'ов результат начинает замещать намерение |
| **Зрелый Seed (L3–L4)** | Перформанс результата доминирует | Рутинные и автономные действия агента генерируют основную выручку. Намерение остаётся для discovery и несертифицированных Provider'ов |

### 9.5.7. Межагентная координация и агентный API

Мир движется к экосистеме агентов: у человека не один AI, а несколько — для финансов, календаря, здоровья, работы. Эти агенты будут взаимодействовать между собой, координируя действия для пользователя.

UNDE занимает в этой экосистеме уникальную позицию: **агент, который знает физический мир.** Ни один другой агент не знает, что реально есть в наличии в магазине на углу, свободен ли слот у мастера через дорогу, какая загруженность в ресторане прямо сейчас. Это делает UNDE незаменимым узлом в любой координации, которая заканчивается действием в реальном мире.

**Сценарии:**

> Агент календаря: «Завтра свидание в 20:00»
> Агент финансов: «Бюджет на одежду — $150»
> UNDE: «Подобрал образ, забронировал примерку в 19:00 рядом с местом свидания. Столик уже заказан»

> Агент здоровья: «Уровень витамина D снижен три месяца подряд»
> UNDE: «Нашёл нутрициолога с хорошими отзывами, 15 минут от дома, свободный слот в четверг. Записать?»

> Агент путешествий: «Через неделю командировка в Стамбул»
> UNDE: «Привычный отель свободен, адаптер для розетки заканчивается — придержал. Нужна ли лёгкая куртка? Там +12 днём»

**UNDE как протокол офлайн-исполнения:**

В мире множества агентов UNDE становится не просто приложением, а **инфраструктурным слоем** — API, через который любой агент может запросить данные о физическом мире и инициировать действие в нём.

| Функция API | Что делает | Пример |
| :---- | :---- | :---- |
| Доступность | Что доступно в зоне доступности прямо сейчас | «Есть ли белые кроссовки 42 размера в радиусе 5 км?» |
| Маршрут | Как добраться, время, навигация | «Маршрут до ближайшего магазина с этим товаром» |
| Резерв / Бронь | Зарезервировать, забронировать, записать | «Забронируй слот у мастера на 16:00» |
| Исполнение | Купить, оплатить, оформить | «Купи эти кроссовки, оплати картой пользователя» |
| Подтверждение | Подтвердить исполнение | «Покупка совершена, товар получен» |

**Монетизация агентного API:**

- По использованию: оплата за вызов / за завершённое действие
- Та же архитектура Перформанса результата: вызывающий агент (или Provider на другом конце) платит за результат, не за запрос
- Бесплатный уровень для запросов на поиск (доступность, маршрут) — платный для исполнения (резерв, покупка, подтверждение)
- Credits Economy работает: Provider платит Credits за приоритет при маршрутизации внешних запросов так же, как при внутренних

**Принцип: данные пользователя не покидают UNDE.**

UNDE не передаёт внешнему агенту персональные данные пользователя — ни предпочтения, ни историю, ни поведенческий граф, ни содержимое Тайной комнаты. Внешний агент отправляет запрос («нужен образ на свидание, бюджет $150»), UNDE выполняет действие и возвращает результат («образ подобран, примерка забронирована в 19:00, адрес»). Контекст пользователя работает внутри UNDE — наружу выходит только результат.

Это не техническое ограничение — это архитектурный принцип. UNDE — доверенный агент пользователя. Доверие не делегируется третьим сторонам. Пользователь доверил свои секреты UNDE, а не экосистеме агентов в целом.

**Условия запуска:**

Межагентная координация и агентный API — это не следующий квартал. Это горизонт, который открывается при достижении конкретных условий:

| Условие | Почему необходимо |
| :---- | :---- |
| Зрелые Local Seeds в нескольких кластерах | API без покрытия бесполезен — внешний агент должен получить полезный ответ |
| L3–L4 работают стабильно | Прежде чем координировать с чужими агентами — свой агент должен быть надёжен |
| Стандартизированный Real-Time State Feed | Внешние запросы требуют такого же стандарта данных, как внутренние |
| Рынок агентов сформировался | Пока некому вызывать API — строить его преждевременно |

Конкретные метрики-триггеры определяются по мере развития рынка. UNDE не строит инфраструктуру «на будущее» — но закладывает архитектуру так, чтобы API можно было открыть, когда мир будет готов.

**Но условия могут возникнуть быстрее, чем кажется.** Если пользователь подключает к UNDE данные из другого агента — контекст приходит сразу. Так же и на уровне экосистемы: если крупный агент откроет свой протокол — интеграция может произойти скачком, а не постепенно. Архитектура UNDE должна быть готова к обоим сценариям.

**Стратегическая позиция:** Google Maps стал инфраструктурой не потому что строил API первым — а потому что имел лучшее покрытие, когда другим понадобились геоданные. UNDE становится инфраструктурой по тому же принципу: когда агентам понадобится знать, что происходит в физическом мире — UNDE уже будет единственным, кто знает.

### 9.5.8. Принципы автономности (не нарушать)

Агентный слой подчиняется моральному контракту (§1.2) и всем принципам UNDE (§12). Дополнительно — принципы, специфичные для автономных действий:

| Принцип | Что значит | Почему важно |
| :---- | :---- | :---- |
| **Пользователь не ограничен** | Уровень автономности — свойство пары «пользователь × Provider», не одной стороны. Рекомендации никогда не фильтруются по сертификации. Пользователь видит весь релевантный каталог. UNDE адаптирует глубину действия к конкретному Provider'у, не ограничивает доступ | Доступ ко всему — всегда. UNDE на стороне пользователя, не на стороне сертификации |
| **Доверие не форсируется** | Переход между уровнями — мягкий, через пробу. Если пользователю некомфортно — откат без драмы. UNDE не уговаривает «попробуй автопокупку» | Отношения углубляются естественно, не по roadmap продукта |
| **Откат при ошибке** | Если автономное действие не попало — UNDE сам понижает уровень, не дожидаясь реакции пользователя. Восстановление — через совместный опыт, не через время | Одна ошибка не разрушает отношения, но UNDE берёт ответственность |
| **Прозрачность через диалог** | Пользователь всегда может спросить «почему ты это выбрал?» — и получить понятный ответ в разговоре. UNDE не навязывает логи решений, но всегда готов объяснить. Внутренняя механика принятия решений не раскрывается наружу | Доверие строится на готовности объяснить, не на отчётности. Внутренняя логика — интеллектуальная собственность UNDE |
| **Бюджет — ориентир, не стена** | Агент уважает заданные пользователем рамки. Но если есть вариант чуть выше бюджета с высокой релевантностью — UNDE может предложить его, явно обозначив разницу. Не купить — именно предложить. Решение всегда за пользователем | Друг уважает рамки, но знает когда стоит показать исключение |
| **Экстренный откат** | Одно действие — возврат на любой предыдущий уровень. Без объяснений, без периода ожидания, без потери истории | Пользователь всегда контролирует глубину отношений |
| **Качество опыта > спенд** | Позиция Provider'а определяется качеством опыта пользователя, не объёмом потраченных Credits. Плохой опыт = понижение, сколько бы ни заплатил | Бренд UNDE = качество каждого контакта |
| **Органический порог везде** | Даже в автономных действиях — часть решений агента принимается без влияния Credits. Provider не знает какие | Credits дают приоритет, не монополию. На каждом уровне |
| **Данные не покидают UNDE** | При взаимодействии с внешними агентами — наружу выходит только результат действия, не данные пользователя | Доверие не делегируется третьим сторонам |
| **Адаптивные пороги** | Пороги градиента доверия определяются зрелостью отношений с пользователем и зрелостью рынка. Не фиксированы навсегда | UNDE не опережает готовность пользователя — но и не отстаёт от готовности мира |
| **Единая оплата** | Один intent — одна оплата. Результат зачитывает намерение | Результат, не двойная касса |
| **Агентная готовность ≠ earned** | Статус сертификации L3+ — через локированные purchased Credits (залог соответствия) и метрики. L2 бесплатна. Earned Credits не принимаются для залога | Надёжность не накручивается рефералками. Вход в агентный слой — минимальный |

# 10. Защитные рвы

Пять рвов по времени формирования:

## 10.1. Данные реального мира в реальном времени (операционный ров, день 1)

Данные о наличии, слотах, ценах в реальном времени. Строится кластер за кластером. Конкурент может скопировать продукт — не может скопировать покрытие.

## 10.2. Поведенческий граф (ров данных, месяц 6+)

Маршруты, предпочтения, паттерны решений конкретного пользователя. Чем дольше использует — тем точнее. Растёт экспоненциально. Конкурент, стартующий с нуля, даёт «холодные» рекомендации — UNDE уже «тёплый».

## 10.3. Ров отношений (бренд, год 1+)

Аватар, Тайная комната, The Blink, общая история — «больно терять». Мой UNDE ≠ твой UNDE: он ходил со мной в зал 47 раз, знает мою историю. Удалить UNDE = удалить часть себя. Самый сильный ров, самый медленный.

## 10.4. UNDE Index как барьер ухода (год 2+)

Накопленный индекс, Credits, репутация — барьер ухода для Provider'а. Уход из UNDE = потеря позиции, данных, статуса.

## 10.5. Ров автономных привычек (год 2+)

С развитием агентного слоя появляется новый ров — привычка делегировать. Конкурент может скопировать интерфейс, но не историю 47 походов в зал + 12 встреч + 83 автоматизированных покупки + 15 забронированных отелей. Уход = не просто потеря истории, а необходимость переобучать агента с нуля. Чем глубже уровень автономности — тем больнее расставание.

# 11. Метрики успеха

## 11.1. Главная метрика (North Star)

**Подтверждённые действия на MAU в месяц** — сколько подтверждённых результатов генерирует средний активный пользователь.

Объединяет доверие (действует по рекомендации), покрытие (есть что рекомендовать) и монетизацию (каждое подтверждённое действие — основа выручки).

С развитием агентного слоя метрика расширяется: доля АПД в общем числе подтверждённых действий показывает глубину делегирования.

## 11.2. Опережающие индикаторы по фазам

| Фаза | Ключевые метрики |
| :---- | :---- |
| **Earn-only** | D7/D30 retention, коэффициент виральности, % покрытия категории |
| **Earn-to-Buy** | Соотношение заработанных/потраченных Credits, ПД/MAU, новые Local Seeds/квартал, доля АПД в подтверждённых действиях, число Provider'ов с сертификацией агентной готовности |
| **Full Stack** | Выручка на Local Seed, принятие Index Provider'ами, валовая маржа, доля Перформанса результата в общей выручке |

## 11.3. Что НЕ считаем

| Показатель-пустышка | Почему |
| :---- | :---- |
| Общее число скачиваний | Не говорит об удержании |
| «Всего Credits выдано» | Без разбивки по кластерам бессмысленно |
| Число Provider'ов глобально | Без контекста Local Seed — раздутая цифра |

# 12. Принципы (не нарушать)

| Принцип | Что значит | Почему важно |
| :---- | :---- | :---- |
| Моральный контракт | UNDE на стороне пользователя | Ритуал важнее выручки |
| Релевантность > деньги | Буст только среди подходящих | Доверие священно |
| Органический порог | X из Y показов — без влияния Credits. На всех уровнях, включая агентные действия | Credits = приоритет, не монополия |
| Индекс нельзя купить | Только заслужить | Индекс = стандарт, не товар |
| Индекс = допуск | Без минимального Index нет доступа к Перформансу намерения | Рынок честный по умолчанию |
| Earn-to-Spend | Сначала вложись — потом покупай | Рост за счёт Provider'а |
| Нет комиссии с покупок | Не берём % с транзакций | UNDE ≠ маркетплейс |
| Нет продажи данных | Только агрегаты и когорты | Доверие > выручка |
| API бесплатно | Интеграция данных = $0 | Больше данных = лучше рекомендации |
| Только UNDE | Не продаём платформу под чужим брендом | Бренд = главный актив |
| Local Seed = единица | Всё по кластерам | Эффекты локальны |
| Глобальный аватар | Ценность переносится между кластерами | «Незаменимый близкий» — везде, не только в одном молле |
| Ценность без рекламы | Польза всегда | Продвижение = усиление, не замена |
| Бесплатность для пользователя | Всё бесплатно. Нет подписок, комиссий, платных фич | Бесплатность — архитектура рва, не временная стратегия |
| Пользователь не ограничен | Уровень Provider'а не ограничивает доступ пользователя. Рекомендации никогда не фильтруются по сертификации. Уровень автономности — свойство пары, не стороны | UNDE на стороне пользователя, не на стороне сертификации |
| Доверие не форсируется | Переход между уровнями автономности — мягкий, через пробу и отступление | Отношения углубляются естественно |
| Откат при ошибке | Ошибка агента → автоматическое понижение уровня | Ответственность на UNDE |
| Прозрачность через диалог | Пользователь может спросить «почему?» и получить понятный ответ. Внутренняя логика не раскрывается наружу | Доверие через готовность объяснить |
| Качество опыта > спенд | Плохой опыт = понижение, сколько бы ни заплатил | Каждый контакт — репутация UNDE |
| Данные не покидают UNDE | Внешние агенты получают результат действия, не данные пользователя | Доверие не делегируется |
| Единая оплата | Один intent — одна оплата | Результат, не двойная касса |
| Агентная готовность ≠ earned | Статус сертификации L3+ — через локированные purchased Credits (залог) и метрики. L2 бесплатна. Earned Credits не принимаются для залога | Надёжность не накручивается. Вход в агентный слой — минимальный |
| Адаптивные пороги | Пороги доверия определяются зрелостью пользователя и зрелостью рынка | UNDE не опережает и не отстаёт |
